# `/goal` 계약 & goal_command 작성법

goalgoal의 산출물은 **Claude Code의 `/goal` 명령**이 소비한다. `/goal`의 실제 동작을 모르고 JSON을 만들면 헛돈다. 이 문서가 그 계약의 단일 진실 원천이다. (출처: https://code.claude.com/docs/en/goal)

## `/goal`은 어떻게 동작하는가

- `/goal <조건>` — 인수는 **자연어 완료조건 문자열**이다. JSON 스키마가 아니다.
- 조건을 설정하면 **즉시 턴이 시작**되고, 조건 자체가 지시문이 된다.
- **매 턴이 끝날 때마다 별도 평가자(빠른 모델, Haiku 계열)가 조건 충족 여부를 yes/no로 판정**한다. "no"면 Claude가 다음 턴을 또 시작하고, "yes"면 goal이 자동 해제된다.
- 세션당 활성 goal은 **1개**. 새 `/goal`은 기존 것을 대체한다.
- `-p`(비대화) 모드에서도 동작하며 루프를 끝까지 돌린다. `Ctrl+C`로 중단.
- 별칭: `/goal clear`(= stop/off/reset/none/cancel)로 해제. `/clear`나 세션 종료로도 해제. `--resume`/`--continue` 시 활성 goal은 복원되나 턴 카운트·타이머는 리셋.
- `disableAllHooks` 환경에선 `/goal` 사용 불가(Stop hook 래퍼이기 때문).

## 결정적 제약 3가지 (goal_command 설계의 근거)

1. **평가자는 도구를 못 돌리고 파일을 직접 못 읽는다.** 오직 **대화(transcript)에 드러난 내용**으로만 판정한다.
   → 모든 완료 기준은 **Claude가 출력으로 증명할 수 있는 것**이어야 한다. 이것이 "**demonstrable-in-conversation**" 철칙이다.
   - ❌ "사용자가 만족하면" · "보기 좋으면" · "성능이 개선되면"
   - ❌ "파일 X를 읽어 확인" (평가자는 못 읽음)
   - ✅ "`swift build`를 실행해 출력이 exit 0" · "`npm test` 통과율 100%가 출력에 표시" · "`ls ~/Desktop \| wc -l`이 10 이하"
2. **조건 문자열은 ≤ 4,000자.** 초과하면 안 된다.
3. **루프가 무한히 돌 수 있다.** 종료 안전장치를 **조건 문장 안에** 넣어야 한다. 예: `"... 또는 20턴 후 멈춘다 (or stop after 20 turns)"`, `"검증이 3회 연속 실패하면 자체 수정을 멈추고 사람의 결정을 기다린다"`.

## goal_command 작성 템플릿

`goal.json`의 `goal_command` 필드(운영 1순위)는 아래 골격을 따른 **산문**이다. `/goal @goal.json`으로 인라인되거나 그대로 복사해 `/goal`에 붙여도 동작해야 한다.

```
<측정 가능한 단일 종료상태>가 될 때까지 멈추지 말고 <목표>를 수행한다.

[증명 방법] 각 마일스톤이 끝나면 다음을 실행해 결과를 출력에 남긴다:
- <체크1: 대화로 증명되는 명령/관찰. 예: `swift build` exit 0>
- <체크2: 예: `ls ~/Desktop | wc -l` 값이 10 이하>

[제약] 다음은 바뀌면 안 된다:
- <제약1. 예: 파일 이동은 반드시 사용자 승인 클릭 후에만>
- <범위 밖 항목은 구현하지 않는다>

[종료 안전장치]
- 위 종료상태가 충족되면 완료로 간주한다.
- 또는 N턴 후 멈춘다 (or stop after N turns).
- 검증이 3회 연속 실패하거나 요구사항이 충돌하면, 자체 수정을 멈추고 사람의 결정을 기다린다.
```

> `N`은 `loop_config.max_iterations`에서 가져온다. 종료상태(완료)와 종료 안전장치(턴 바운드·실패 시 중단)는 **반드시 둘 다** 들어가야 한다 — 전자는 `done_definition`, 후자는 `loop_config.stop_condition`에서 합성한다.

### 좋은 예 (clean-desktop)

```
바탕화면 정리 기능 4개가 'AI 제안 → 사용자 승인 → 정리' 플로우로 동작하고 빌드가 통과할 때까지 멈추지 말고 macOS 메뉴바 앱을 구현한다.

각 마일스톤이 끝나면 다음을 실행해 결과를 출력에 남긴다:
- `swift build`가 exit 0으로 통과
- 앱 실행 후 메뉴바 항목 4개(바탕화면/다운로드/스크린샷/중복찾기)가 로그에 표시
- 정리 시뮬레이션에서 사용자 승인 없이 이동된 파일 수가 0으로 출력

다음 제약은 바뀌면 안 된다:
- 파일 이동·삭제는 반드시 사용자 승인(클릭) 후에만 실행한다.
- 화면·집중 세트, 클립보드 세트, 클라우드 동기화는 구현하지 않는다.

종료 안전장치: 위가 모두 충족되면 완료. 또는 25턴 후 멈춘다 (or stop after 25 turns). 빌드가 3회 연속 실패하면 자체 수정을 멈추고 사람의 결정을 기다린다.
```

## 다른 Loop 타깃 (참고)

사용자가 `/goal`이 아닌 다른 루프를 원하면 매핑만 바꾼다. 핵심 필드는 동일하게 재활용된다.

| 타깃 | 무엇을 먹나 | goalgoal 필드 매핑 |
|------|-------------|--------------------|
| **Claude Code `/goal`** (기본) | 자연어 완료조건 문자열 | `goal_command` 그대로 |
| ouroboros seed | seed JSON (설치 버전별 스키마 상이) | `goal`→objective, `success_criteria`→acceptance, `loop_config`→limits. **변환 시점에 설치된 ouroboros 스키마를 확인**할 것 |
| `/loop <프롬프트>` | 반복 프롬프트 | `goal_command`를 프롬프트로, 종료는 외부 interval로 |

ouroboros seed 정확한 스키마는 버전마다 달라지므로 하드코딩하지 않는다. 변환 요청 시 설치된 플러그인의 seed 예시를 먼저 확인하고 매핑한다.
