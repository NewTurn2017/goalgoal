# 마스터 질문 풀 (20개) + 운영 규칙

goalgoal 인터뷰의 단일 진실 원천. 20개를 다 묻지 않는다 — 도메인과 이미 또렷한 축에 따라 `skip_if`로 가지치기하여 **실제 8~16개**만 4개씩 묶어 AskUserQuestion으로 호출한다.

## 묶음 구성 원칙

| 묶음 | 구성 | 비고 |
|------|------|------|
| 1차 (필수) | A그룹 4개 | 도메인 무관. 절대 생략하지 않음 |
| 2차 | C그룹(도메인) 2 + D그룹 검증 1 + D그룹 범위/리스크 1 | 도메인 분류 결과로 C 선택 |
| 3차 (조건부) | B그룹 잔여 + 모호한 축 보강 | done_definition/constraints가 아직 모호할 때만 |

각 질문은 옵션 최대 4개 + "잘 모르겠어요" + (자동 제공되는 Other 자유입력). 질문 머리말에 진행률 "n/N"을 넣는다.

---

## A그룹 — 목표 & 결과물 (필수 4, skip 불가)

- **Q1 핵심 목표**: 한 문장으로, 뭘 만들거나 이루고 싶은가? → `goal`
- **Q2 결과물 형태**: 앱 / CLI 도구 / 자동화 스크립트 / 문서·리포트 / 데이터·모델 → `domain` 힌트
- **Q3 사용자·맥락**: 나만 / 팀 / 불특정 다수, 어디서 쓰나 → `context`
- **Q4 규모·야심**: 빠른 실험(spike) / 최소 동작(MVP) / 운영급(production) → `scale`, `loop_config.max_iterations` 기본값 결정

## B그룹 — 성공 & 종료 (필수 4, 일부는 3차로 이월 가능)

- **Q5 성공 측정 방식**: 성공을 무엇으로 재나? **대화 출력으로 증명되는** 기준이어야 함 (숫자·%·통과/실패·`exit 0`·"메뉴바에 표시"). "사용자가 만족" 같은 확인 불가 기준은 금지 — `loop-formats.md`의 demonstrable 철칙 참조 → `success_criteria`
- **Q6 완료 상태(done)**: "완전히 끝났다"는 어떤 상태인가? → `done_definition`
- **Q7 강제 중단(stop)**: 루프를 억지로라도 멈춰야 할 때는? (반복 한계·오류·시간) → `loop_config.stop_condition`
- **Q8 최대 반복 감각**: 몇 번쯤 돌면 충분/과함? → `loop_config.max_iterations`

> Q6와 Q7은 사용자에게 **하나의 질문**("성공적으로 끝났을 때 vs 강제로 멈춰야 할 때")으로 물어도 되며, JSON 변환 시 둘로 분리한다.

## C그룹 — 도메인 특화 (8 후보 중 도메인별 2 선택)

도메인 분류 결과에 따라 아래에서 2개를 골라 2차 묶음에 넣는다.

| 도메인 | 후보 질문 (택2) |
|--------|----------------|
| code | 언어/프레임워크? · 테스트 포함 여부? · 기존 코드베이스 위 vs 새로? · 배포 타깃? |
| data / analysis | 입력 소스·형식? · 출력 포맷? · 데이터 규모? · 정확도 기준? |
| automation | 트리거(수동/스케줄/이벤트)? · 외부 서비스 연동? · 실패 시 알림·재시도? · 멱등성 필요? |
| creative | 톤/스타일? · 분량·개수? · 참고 레퍼런스? · 검수 기준? |
| infra | 환경(클라우드/로컬)? · 롤백 전략? · 비용 한도? · 멱등성/idempotency? |

`skip_if`: 앞 답변에서 이미 드러난 항목(예: Q1에서 "Python FastAPI"라고 했으면 언어/프레임워크 질문 생략)은 묻지 않는다.

## D그룹 — 범위 & 리스크 & 검증 (4)

- **Q17 범위 밖(out_of_scope)**: 이번엔 **안** 할 것은? → `out_of_scope`
- **Q18 제약(constraints)**: 시간/비용/기술/금지사항 → `constraints`
- **Q19 최대 리스크**: 가장 크게 어긋날 수 있는 부분·실패 모드 → `risks`
- **Q20 검증 방법(verification)**: 각 반복 결과를 무엇으로 확인하나? (자동 테스트 / 사람 리뷰 / 메트릭) → `verification_method`

## max_iterations 도메인 기본값

| 도메인 · 규모 | 기본 범위 |
|---------------|-----------|
| code · spike | 5~10 |
| code · mvp | 15~30 |
| code · production | 30~60 |
| data / analysis | 5~15 |
| automation | 5~20 |
| infra | 3~8 |
| creative | 2~5 |

사용자가 Q8에 "잘 모르겠어요"면 이 기본값을 쓰고 `confidence`를 한 단계 낮춘다.

## 인터뷰가 안 묻고 자동 처리하는 것

| 항목 | 처리 |
|------|------|
| `risk_tier` | **묻지 않는다.** Step 1에서 LLM이 답변·도메인으로 판정 (none/low/elevated). 파일 변경·시스템 접근·사용자 콘텐츠 읽기·외부 전송 신호가 있으면 elevated → `safety` 블록을 인터뷰 막바지에 보강 질문하거나 답변에서 합성 |
| `safety` | risk_tier=elevated일 때만. 대부분 기존 답변(승인 게이트·제약·리스크)에서 합성. 권한·프라이버시가 불명확하면 2차/3차에 1문항 추가 |
| `goal_command` | **묻지 않는다.** Step 7에서 다른 필드를 `loop-formats.md` 템플릿으로 합성. 측정가능 종료상태 + demonstrable 체크 + 제약 + 턴바운드(=max_iterations) + 3회 실패 시 사람 대기 |

## 운영 규칙 (Step 4가 따른다)

### Other(자유입력) 재앵커링
"Other"로 비결정적 텍스트("둘 다", "경우에 따라")가 오면, 다음 묶음 첫 질문을 **"말씀하신 내용이 이 4가지 중 어디에 가깝나요?"**로 교체해 다시 좁힌다. 그래도 안 좁혀지면 `goal.json`의 `freetext_note`에 원문을 보존하고 해당 정형 필드는 미정으로 둔다.

### 조기종료 — 필수 4축 커버리지 (raw 카운트 아님)
"잘 모르겠어요"를 **몇 번** 골랐는지 세지 않는다. 사소한 도메인 질문과 핵심 기준을 동급 취급하면 안 되기 때문이다. 대신 **필수 4축**(목표 · 성공기준 · 종료조건 · 범위) 중 **2개 이상이 미해결**로 남으면:
1. 남은 축을 도메인 기본값(sensible defaults)으로 채우고,
2. 채운 필드를 `open_questions`에 기록하고,
3. `confidence: "low"` + `status: "needs_review"`로 세팅한 뒤 조기 종료한다.

4축이 모두 또렷하면(모름이 사소한 질문에만 몰려도) 정상 종료해도 된다.

### 모순 검사
서로 배타적인 답변 조합(예: `mobile_app` + `cli_target`, `spike` + `max_iterations: 100`)을 감지하면 종료 전 재확인 질문을 1개 끼운다. 사용자가 고른 쪽으로 다른 쪽을 override한다.

### 조기 종료
인터뷰 도중 4축(목표·성공기준·종료조건·범위)이 모두 또렷해지고 모순이 없으면, 남은 묶음을 생략하고 Step 7로 진행한다 — 피로도를 줄이는 핵심 장치다.
